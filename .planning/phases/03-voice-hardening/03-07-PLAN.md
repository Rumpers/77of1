<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Fan browser ↔ /api/voice/:jobId | Untrusted client; HMAC token is the access control |
| HMAC token in URL ↔ Replit Secrets | URL is potentially logged (request logs, browser history); token must time out |
| generation_jobs.result_url ↔ proxy route | Storage-key derivation must NOT be guessable from outside |
| Object Storage object ↔ proxy stream | Stream copy must not amplify request size for DOS |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-07-01 | Spoofing | Attacker guesses jobId + forges token to access another fan's voice | mitigate | HMAC token bound to jobId + exp; signing secret stays server-side; UUID jobIds = 122 bits of entropy. Brute force infeasible |
| T-03-07-02 | Information disclosure | Voice URL leaks via referer/browser history → another party plays it | mitigate | 24h TTL (configurable); content is non-PII transcript; SB 243 disclosure footer makes intent unambiguous. Cache-Control: private, max-age=0 prevents CDN caching |
| T-03-07-03 | Tampering | Modified token bypasses verification | mitigate | timingSafeEqual + length check; production secret is 64-char hex (256-bit). Test in voice-token.test.ts |
| T-03-07-04 | Information disclosure | generation_jobs.result_url contains storage key; if leaked via API, attacker can probe other paths | accept | Storage key includes creatorId UUID; even with key, fetch requires bucket auth. Defense-in-depth via the proxy means clients never see storage keys directly |
| T-03-07-05 | Denial of service | Many invalid-token requests overload the proxy | accept | Token verify is sub-millisecond; bad tokens 403 early before DB query. Add rate limiting in v2 if necessary |
| T-03-07-06 | Elevation of privilege | Cross-creator voice URL reuse — same jobId → different fan accesses it | mitigate | jobId scopes a single generation_jobs row → one creatorId. Per-fan binding via fanIdHash is not enforced in URL (the voice file IS the deliverable to whoever the original request was for) — this is a stated design choice: voice URLs are bearer tokens with 24h TTL |
| T-03-07-07 | Information disclosure | OpenAPI codegen regenerates and accidentally drops fields | mitigate | codegen is invoked deterministically; the new path is additive; existing endpoints remain. Verify post-codegen by typecheck |
| T-03-07-08 | Tampering | (V13 API Security) Missing input validation on jobId path param | mitigate | UUID_RE regex check on jobId param BEFORE HMAC verify BEFORE DB lookup. Even with token bypass, malformed jobIds 403 |
| T-03-07-09 | Information disclosure | Voice file ID logged in Sentry → trail of who fetched what | accept | jobId in logs is not PII; mapping to fan_id requires DB access. SB 243 audit covered separately by safety_audit_log |
</threat_model>
