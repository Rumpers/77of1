<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Hermes Telegram identity ↔ /dsar command | Telegram user_id → creator_id mapping via `findCreatorByTelegramId` |
| BullMQ job.data ↔ worker | Worker receives `creatorId` from queue; cannot assume it's authentic without cross-check |
| Object Storage prefix `creators/{id}/` ↔ DSAR sweep | Deletion sweep must scope to this prefix only — never touch other creators' data |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-04-01 | Spoofing | Attacker with Redis access enqueues `{type:"dsar-deletion", creatorId:"<victim>"}` directly | mitigate | Task 3.6: worker cross-checks `creator_deletion_log.auditId` exists with matching auditId before deleting. Hermes wizard is the only writer of that row, and it requires authenticated Telegram identity. Without a pre-existing audit row, the worker aborts |
| T-03-04-02 | Tampering | DSAR replay (re-running same auditId) re-runs the sweep | mitigate | BullMQ `jobId: auditId` deduplicates at queue level (RESEARCH §Threat Patterns); the worker's "audit row exists" check tolerates re-runs idempotently |
| T-03-04-03 | Repudiation | Creator denies they requested DSAR | mitigate | `creator_deletion_log` row carries `auditId = sha256(creatorId.requestedAt).slice(0,16)` — sent to creator at confirmation time; creator-provided audit ID matches stored row |
| T-03-04-04 | Information disclosure | Sweep leaves orphan rows containing fan-id-hash references to deleted creator | accept | All in-scope tables FK-cascade or are explicitly cleared. Telegraf_sessions is NOT swept here (no creator_id FK); covered by Phase 2's `@telegraf/session/pg` retention policy |
| T-03-04-05 | Denial of service | Object Storage list returns 50k+ objects, sweep hangs the worker | mitigate | Concurrency: 1 already serializes; add SLA warn-log at 60s. Object count per creator is bounded (~1 reference + N voice generations); 50k case implies a separate bug |
| T-03-04-06 | Elevation of privilege | DSAR sweep accidentally hits the wrong creator's data | mitigate | All deletes scoped by `eq(table.creatorId, payload.creatorId)`. Object Storage prefix scoped to `creators/${creatorId}/`. Both come from the BullMQ payload which the audit-row check (T-03-04-01) authenticates |
| T-03-04-07 | Tampering | Sweep aborts mid-flight leaving inconsistent state | mitigate | Order: deletes first → twin anonymize → creators anonymize → audit row LAST. On failure, `completedAt = NULL` triggers retry. No partial-complete reports |
| T-03-04-08 | Tampering | (Pitfall 8 — race) Fan-twin processes a chat turn between DSAR confirmation and 24h sweep, recreating data | mitigate | Task 2: `setKillSwitchActive(creatorId, true)` runs BEFORE enqueue; KYC-gate middleware (Phase 1) returns 423 on `kill_switch_active=true` — no new chat data can be created during the grace window |
| T-03-04-09 | Information disclosure | Creator's preferred language inferred → wrong locale on confirmation | accept | Falls back to 'en'; creator can re-issue /dsar |
</threat_model>
