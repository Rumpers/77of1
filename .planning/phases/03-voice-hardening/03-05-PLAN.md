<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Telegram user_id ↔ founder gate | Only allow-listed Telegram user_ids can run /review_masks; everyone else gets the unauthorized message |
| callback_data string ↔ DB UUID lookup | Telegram callback_data is attacker-controllable (anyone with a forwarded message can craft it); MUST regex-validate before DB query |
| Inline keyboard from approval row ↔ subsequent state | An approved row should never re-appear in /review_masks; setMaskReviewed enforces this |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-05-01 | Elevation of privilege | Non-founder runs /review_masks and approves masks | mitigate | Task 2.B.2: `isFounder()` gate on both the scene-entry command AND the callback handler. FOUNDER_TELEGRAM_USER_ID env var lists allowed user_ids |
| T-03-05-02 | Tampering | Forged callback_data like `mask:approve:'; DROP TABLE--` | mitigate | UUID regex `/^[0-9a-f]{8}-...$/i` validation in BOTH index.ts (Task 2.B.4) and db.ts (Task 1.A.2); parameterized SQL prevents injection regardless |
| T-03-05-03 | Information disclosure | Mask review shows raw fan PII (the candidate string IS the name) to a non-founder | mitigate | Founder-gate on the scene entry blocks fetch before reply. UUID regex blocks abuse via callback_data |
| T-03-05-04 | Repudiation | Founder denies approving a mask | accept | reviewed_at + approved are persisted; correlate with Telegram chat history. No additional logging needed for N=1 |
| T-03-05-05 | Denial of service | Empty queue spam (/review_masks repeated) | accept | getNextPendingMask is a single indexed lookup (partial index from 03-02). No rate limit needed |
| T-03-05-06 | Information disclosure | Mask candidates leaked in console logs | mitigate | Do NOT console.log mask candidate text in the bot.action handler; only log the decision + id |
</threat_model>
