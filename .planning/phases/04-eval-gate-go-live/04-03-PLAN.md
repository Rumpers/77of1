<objective>
Build the go-live gate as a founder-authed admin endpoint AND wire the runtime gate it controls.
POST /api/admin/twin/:creatorId/activate reads the latest eval run via isGoLiveEligible and only
flips twins.status to "active" when the hard-limit and injection categories both passed 100%. To
make that flag actually mean something, this plan also adds the missing `twins.status === 'active'`
check to the fan chat route — without it, the activate endpoint would toggle a flag nothing reads.

Purpose: EVAL-01 success criterion 1 — the twin cannot go live unless hard-limit and injection
scores are both 100%. VERIFIED FACT (BLOCKER-01): routes/twin.ts currently has NO twins.status
check; the only runtime gates are creators.kill_switch_active and creator_config.paused (both →
503 creator_paused), plus KYC + credits. twins.status is only ever set to "deleted" by the DSAR
worker, and new twins default to "inactive" (schema default; hermes INSERT at db.ts:296 sets no
status). So this plan establishes twins.status='active' as a real runtime gate (greppable, matches
the eval-gate mental model) and makes the activate endpoint the controlled transition into it.

Output: a runtime twins.status gate in the chat route, a founder-auth middleware, the activate
route wired into the api-server router, and tests proving the 401 / 422 / 200 paths plus the
inactive-twin 503.
</objective>
